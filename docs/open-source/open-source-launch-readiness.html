Executive assessment

The repository is suitable for private launch preparation. No historical secrets were detected. The application already uses a user-owned OpenRouter credential stored in Expo SecureStore, and provider error handling is designed not to expose token or transcript payloads.

Publication remains blocked intentionally until the repository rename, Vercel Git integration, security settings, final public-content verification, and visibility change are explicitly confirmed.

Known dependency advisory

Moderate tooling advisory

uuid@7.0.3 is introduced by expo → @expo/config-plugins → xcode. The advisory concerns buffer bounds in UUID v3/v5/v6 APIs when a caller supplies a buffer. SayCopy does not call that package; it is in the Node-based Expo/Xcode configuration path rather than the mobile application's runtime request path.

The registry's automated fix proposes replacing Expo 57 with Expo 46, a breaking and unsafe downgrade. Do not run npm audit fix --force. Continue using Expo-supported releases, let Dependabot report upstream resolution, and reassess when Expo or xcode updates the dependency.

Main-branch protection after CI is green

  • Require pull requests for main.
  • Require the CI / quality check and require branches to be up to date.
  • Require conversation resolution and linear history.
  • Block force pushes and branch deletion.
  • Use zero mandatory outside approvals initially if this remains a single-maintainer repository; increase to one approval when another regular reviewer is available.

GitHub currently returns a plan/visibility restriction for branch protection while the repository is private. Apply the rule immediately after the repository becomes public, when the feature becomes available.

Vercel Git integration plan

  1. After the repository rename is complete, connect altitudeinfosys/saycopy to the existing Vercel project saycopy.
  2. Set the project root directory to website and leave framework detection as Other/static.
  3. Set main as the production branch. Allow automatic previews for pull requests and non-production branches.
  4. Keep RESEND_API_KEY, CONTACT_TO_EMAIL, and CONTACT_FROM_EMAIL production-only. Do not expose the production Resend key to preview builds from contributor code. A preview without mail credentials should return the existing safe 503 response.
  5. Use CONTACT_ALLOWED_ORIGINS only for additional controlled origins. Vercel preview origins are derived from VERCEL_URL.
  6. Protect main with CI so production deploys originate only from merged, passing commits. Treat the Vercel deployment status as an additional required check after the integration creates a stable check name.
  7. After connection, verify one pull-request preview, merge it, verify the production alias and saycopy.app, then retire manual CLI deployment as the normal path.

The contact function's in-memory rate limit is process-local and should not be treated as a durable anti-abuse control. Before sustained public traffic, configure Vercel Firewall/rate limiting or another durable shared limiter as an account-side follow-up.
