How To Build The App

Use this path for a fresh Mac or after pulling new changes. Until we create our own signed DMG, building locally from the private repo is the most reliable install path. The build still outputs VoiceInk.app; the Local Flow rename can happen later.

Prerequisites

  1. Make sure the Mac has access to the private GitHub repo at https://github.com/altitudeinfosys/VoiceInk-FlowLocal.
  2. Install Xcode or Xcode Command Line Tools.
  3. Install Homebrew if it is not already installed.
  4. Install CMake with Homebrew.
xcode-select --install
command -v brew || echo "Install Homebrew from https://brew.sh first"
brew install cmake

Fresh Clone Build

  1. Clone the private repo from GitHub.
  2. Check out the current working branch.
  3. Run the local build command.
  4. Confirm the app appears in ~/Downloads.
mkdir -p ~/Projects/code
cd ~/Projects/code
git clone --branch flowlocal-v1 https://github.com/altitudeinfosys/VoiceInk-FlowLocal.git
cd VoiceInk-FlowLocal
make local
ls -lh ~/Downloads/VoiceInk.app

Existing Checkout Build

  1. Use this if the repo is already cloned on the Mac.
  2. Pull the latest private branch.
  3. Run make local again.
cd /Users/tarekalaaddin/Projects/code/VoiceInk-FlowLocal
git checkout flowlocal-v1
git pull --ff-only private flowlocal-v1
# If this checkout was cloned directly from the private repo, use this instead:
# git pull --ff-only origin flowlocal-v1
make local
ls -lh /Users/tarekalaaddin/Downloads/VoiceInk.app

Build Output

  • make local builds the app and copies the runnable app bundle to ~/Downloads/VoiceInk.app.
  • After building, install the app by copying ~/Downloads/VoiceInk.app into /Applications.
  • If macOS blocks the first launch, right-click the app in Finder and choose Open.

Manual Test Plan

1. Language Profiles

  1. Launch the app with a fresh or reset onboarding state.
  2. Complete onboarding.
  3. Open mode/profile settings and confirm English is active/default.
  4. Confirm English uses Parakeet V3 parakeet-tdt-0.6b-v3, language en, OpenRouter cleanup, and the English Medium Cleanup prompt.
  5. Switch to Arabic and confirm it uses scribe_v2, language ar, and Arabic Light Cleanup.
  6. Switch to Spanish and confirm it uses scribe_v2, language es, and Spanish Light Cleanup.

Expected: English is local-first. Arabic and Spanish are cloud-first. Old no-AI Dictation is not the final onboarding default.

2. English Dictation Path

  1. Select the English profile.
  2. Place the cursor in Notes, TextEdit, or another editable app.
  3. Start recording with the app hotkey.
  4. Dictate a short English sentence with minor filler words.
  5. Commit with Return or the configured hotkey.

Expected: text is transcribed locally, cleaned by the configured cleanup model if keys are configured, and inserted at the cursor.

3. Arabic and Spanish Cloud Paths

  1. Add/confirm ElevenLabs API key.
  2. Select Arabic profile and dictate a short Arabic phrase.
  3. Confirm the menu bar cloud badge appears when the Arabic profile is active and usable.
  4. Repeat with Spanish.

Expected: cloud transcription runs through Scribe v2. Cleanup is light and should not aggressively rewrite wording.

4. Cloud Reminder

  1. Reset the reminder flag if you need to retest: remove HasShownCloudProfileReminder from app defaults, or use a fresh defaults domain.
  2. Select English local profile.
  3. Switch to Arabic or Spanish cloud profile from the menu bar or mode popover.

Expected: a one-time warning appears saying the profile uses cloud transcription. It should not repeat after the first warning.

5. Recorder Controls

  1. Start recording from the recorder panel.
  2. Press Esc.
  3. Start a second recording.
  4. Press Return.
  5. Start a third recording.
  6. Press fn+Space.

Expected: Esc cancels and inserts nothing. Return and fn+Space commit and continue through transcription/cleanup/paste.

6. Review Before Insert

  1. Open Settings -> Pasting.
  2. Enable Review Before Insert.
  3. Start a recording and commit it.
  4. Edit the text in the review panel.
  5. Click Copy and confirm the edited text is copied.
  6. Press Shift+Return inside the editor and confirm it inserts a newline.
  7. Press unmodified Return to commit.
  8. Repeat and press Esc or Cancel.

Expected: review mode pauses before insert. Edited text is what gets pasted. Cancel closes the panel and inserts nothing. Review mode off restores fast direct insertion.

7. Privacy and Retention

  1. Open History Settings.
  2. Confirm transcript cleanup is enabled and retention is 30 days.
  3. Confirm audio cleanup defaults to Temporary and debug recorder audio retention is off.
  4. Record and commit a test phrase.
  5. Check the app recordings directory for a UUID-named recorder .wav file from that session.
  6. Cancel a recording and repeat the file check.
  7. Enable Retain Recorder Audio for Debugging and repeat a short recording.

Expected: committed and canceled recorder audio is deleted by default while transcript text remains. Debug retention keeps recorder audio.

8. Imported Audio Safety

  1. Use the app's audio-file transcription flow with an imported file.
  2. Confirm the transcript is created.
  3. Run normal recorder tests again.

Expected: immediate recorder cleanup only applies to recorder-created UUID .wav files, not imported/manual audio copies such as transcribed_*.wav.

Future Feature Notes

  • Stats effort: low to medium. The app already records sessions, audio duration, words, transcription model, cleanup model, and cleanup duration. The first version should be a read-only page using the existing stats store.
  • Cost tracking effort: medium. The app needs to persist provider request metadata such as model, input tokens, output tokens, audio seconds, provider, and pricing snapshot. Pricing changes over time, so all costs should be labeled estimated.
  • Pricing refresh: medium to high depending on provider support. Start with a local editable pricing table and a manual "Update Pricing" action. Provider APIs can be added later when stable enough.
  • Translate effort on Mac: medium. The simplest version can reuse the existing cloud cleanup/provider path: record -> transcribe -> translate prompt -> show translated text -> copy/insert. A polished version needs language selectors, history metadata, and offline/error states.
  • Mobile effort: medium for a simple iOS app, higher for iOS plus Android. A practical v1 should be cloud-first, with one record button, source/target language selectors, transcript/translation output, copy/share, and database sync. Native keyboard-level insertion is the hard part and should not be v1.
  • OpenRouter selector note: the current OpenRouter detail drawer is a provider-management screen, not the model selector. To choose a model today, open Modes, edit the target mode such as English, choose OpenRouter as the AI provider, then use that mode's AI Model picker. Future UI should make this path discoverable from the catalog itself.

English Online and Offline Flow

When English uses Parakeet V3: Parakeet runs on the Mac and converts recorded speech into raw transcript text. Gemini runs in the cloud and only receives the raw text for cleanup; it does not need the original audio.

Online English:
microphone audio -> Parakeet V3 local transcription -> raw transcript -> Gemini cleanup -> polished text -> insert at cursor
Offline English:
microphone audio -> Parakeet V3 local transcription -> raw transcript -> insert raw text or show cleanup-unavailable prompt

Offline expectation: local English transcription should continue working after the model is downloaded, but Gemini cleanup will not run without internet. The app should make that state obvious and offer to insert the raw transcript instead of failing silently.

Arabic/Spanish expectation: the v1 design keeps Arabic and Spanish cloud-first, so those profiles still require internet for cloud transcription and cleanup.
