Design doc for review

Tarek-Wisper V1

Focused cloud MVP for an iOS-first Expo/React Native app that records short speech clips, transcribes them, optionally translates them, and stores local text history with simple search and tags.

Date: 2026-07-05 Status: Awaiting user design approval Branch: main Latest synced commit: 74727d0 Artifact: HTML

Decision Summary

Approved direction: Approach A, the Focused Cloud MVP. V1 is account-free, iOS-first, cloud-only, OpenRouter-primary, short-clip oriented, and stores text history only on the device.

  • Framework: Expo/React Native with TypeScript, targeting iOS first and keeping Android feasible later.
  • Navigation: three main tabs: Record, History, and Settings.
  • Modes: Record tab contains Transcribe and Translate modes.
  • Recording: tap once to start, tap again to stop, with an approximately 60-second V1 cap.
  • Provider: OpenRouter is the primary provider for speech-to-text, translation, and light cleanup.
  • Languages: English, Spanish, Arabic, and Auto-detect are first-class choices.
  • Translation: single-speaker only in V1. Conversation mode is intentionally deferred.
  • History: successful text results auto-save locally. Audio is temporary and deleted after result creation.
  • Output: result text is editable before copy/share; saved history reflects the edited visible result.
  • Distribution gate: before Expo preview builds, EAS builds, TestFlight, or similar external build steps, stop and tell the user.

Non-Goals

  • No accounts, login, team sharing, backend API, or synced history in V1.
  • No offline or local speech models in V1.
  • No long-audio chunking, diarization, live conversation mode, or continuous dictation in V1.
  • No full OpenRouter model picker in the primary V1 UI.
  • No nested folders, AI auto-tagging, or complex taxonomy management.
  • No embedded shared production OpenRouter token in the app bundle.

Architecture

The app should be built as a small, layered Expo app. UI screens call application services; application services call provider and storage adapters; provider and storage adapters wrap SDKs or platform APIs. This keeps the OpenRouter dependency, audio recording implementation, and local persistence independently testable.

UI Layer

Record, History, Settings, result editor, tag controls, empty states, permission states, and error states.

App Services

Recording session state, transcription flow, translation flow, cleanup flow, history mutations, and settings state.

Adapters

OpenRouter client, audio recorder, secure token store, SQLite history store, clipboard/share wrappers, file cleanup.

Domain Types

Language options, mode options, model presets, history item shape, provider result shape, and typed error categories.

Recommended Module Boundaries

Provider Flow

OpenRouter is used through two stable provider methods: a dedicated speech-to-text call and chat completions for translation and cleanup. The app should keep these behind an internal provider interface so another provider can be added later without rewriting screens.

1
Record

Use expo-audio to record a short clip into app cache. The default output format should be .m4a unless implementation testing shows a better OpenRouter-compatible format.

2
Transcribe

Base64 encode the temporary audio and call /api/v1/audio/transcriptions with model, input_audio, and optional language.

3
Cleanup

In Transcribe mode, send transcript text to chat completions for light cleanup unless the user disables cleanup.

4
Translate

In Translate mode, send the transcript or manual source text to chat completions with source/target language instructions.

5
Save

Delete temporary audio after the result object is created. Auto-save visible text fields to local SQLite history.

OpenRouter Details

Capability Endpoint V1 usage
Speech to text POST https://openrouter.ai/api/v1/audio/transcriptions Use JSON body with base64 audio. Include language for English, Spanish, and Arabic. Omit language for Auto-detect.
Translation POST https://openrouter.ai/api/v1/chat/completions Use low-temperature text prompts that return only translated text. Target language must be explicit.
Light cleanup POST https://openrouter.ai/api/v1/chat/completions Fix punctuation, capitalization, spacing, obvious filler, and small transcription artifacts while preserving meaning.

Model Presets

V1 should expose presets, not a model picker. Presets map to model IDs in a central registry so implementation can update defaults without changing UI logic. Current live OpenRouter catalog candidates checked on 2026-07-05:

Preset Current candidate Use Fallback candidate
Fast google/gemini-3.1-flash-lite Default for quick cleanup and everyday translation when speed/cost matter most. openai/gpt-4o-mini
Balanced openai/gpt-4.1-mini Default recommended preset for V1 if real-device tests show good translation quality and latency. google/gemini-2.5-flash
Best Quality anthropic/claude-sonnet-4.6 User-selected option for harder translation or cleanup where cost and latency are less important. anthropic/claude-sonnet-4.5

The STT model should also be registry-backed. The OpenRouter STT reference currently shows openai/whisper-large-v3 as an example. Final STT default should be verified with real English, Spanish, and Arabic clips during implementation before any readiness claim.

Screens

Record Tab

Record is the primary working surface. It should show a compact mode switch for Transcribe and Translate, language selectors, the recording control, processing state, and the latest editable result. It should not feel like a marketing screen or tutorial.

  • Top controls: mode segmented control, model preset menu, source language selector.
  • Translate-only controls: target language selector and optional manual text input area.
  • Recording control: one large tap target that toggles start/stop and displays elapsed time.
  • Limit state: at about 60 seconds, stop recording and move into processing automatically.
  • Processing state: show stage-specific status such as transcribing, cleaning up, or translating.
  • Result editor: editable text area for final transcript or emphasized translation.
  • Translate result layout: translated output first and visually emphasized; original transcription/source text below.
  • Actions: Copy, Share, edit tags, delete result from history if already saved, retry failed stage where possible.

History Tab

History is a local text archive. It should be optimized for finding recent results quickly, not for managing a document library.

  • Search input filters across primary result text, source text, translated text, language names, and tags.
  • Tag filter row shows user-created tags plus an All state.
  • Chronological list groups items by recent date buckets or simply sorts newest first for V1.
  • Each row shows mode, primary text preview, date/time, language pair when relevant, and tags.
  • Detail view supports editing visible text fields, adding/removing tags, Copy, Share, and Delete.
  • Empty states should be direct: no history yet, no search matches, or no items for a tag.

Settings Tab

Settings should separate secrets, defaults, and product behavior. The user must be able to understand whether the app is ready to call OpenRouter.

  • OpenRouter token: secure entry, save, replace, clear, and present/missing status. Do not display the full saved token.
  • Defaults: mode, source language, target language, model preset, cleanup enabled.
  • Recording: clip length cap shown as V1 behavior; background recording remains disabled in V1.
  • History: local-only explanation, delete all history action with confirmation.
  • About: working name, version, privacy notes, and provider links.

Data Model and Local Storage

V1 should use expo-sqlite for local text history and non-secret settings. It gives durable relational storage for search, tags, edits, and future migrations without requiring a backend. The OpenRouter token should use expo-secure-store, which is built for encrypted local key-value secrets and uses iOS Keychain services.

Persistence Rules

  • Audio files are temporary processing inputs only. Store them in cache and delete after success, cancel, or unrecoverable failure cleanup.
  • History stores text only: user-visible final transcript, original/source text for translation, translated text, language metadata, tags, and timestamps.
  • Do not store hidden raw provider output in V1 history. Intermediate raw text may exist in memory during processing and should be discarded after result creation.
  • If cleanup fails after transcription succeeds, show the raw transcript as the editable result and save that visible text if the result is accepted.
  • Manual translation inputs are text and can be saved as the source text for translation history items.
  • Token is stored only in SecureStore. Do not copy it into SQLite, logs, error messages, or source-controlled files.

Search

V1 search can start with normalized LIKE queries over relevant text fields and tag names. If performance becomes poor with real data, add SQLite FTS in a later migration. Search should be case-insensitive and should match Arabic text as stored; advanced stemming is out of scope.

Error Handling

Errors should be typed in the application layer and displayed as short, actionable messages. The app should preserve user work whenever a partial result exists.

Scenario User-facing behavior Internal handling
No OpenRouter token Record and manual translation actions prompt the user to add a token in Settings. Provider client fails before network request with missing_token.
Microphone permission denied Record button shows permission blocked state and points to system settings when the app cannot ask again. Audio adapter exposes permission status and does not start recorder.
No network or timeout Show retry option. If no transcript exists, do not save an empty history item. Map fetch failures and timeout to network_unavailable or timeout.
Provider authentication or payment error Explain that the OpenRouter token or account needs attention. Map 401, 402, and related error bodies without logging the token.
Rate limit or provider overload Offer retry after a short delay and preserve local audio only until the retry/cancel decision. Map 429, 502, 503, 524, and 529 to retryable provider states where appropriate.
Cleanup failure after STT success Show raw transcript as editable output with a small cleanup-failed notice. Discard audio after creating the text result; save only visible edited text.
Translation failure after STT success Show original transcription and allow retry translation, copy original, or abandon result. Keep text in memory for retry; do not auto-save as a translation until translated output exists.
Audio cleanup failure No prominent user interruption unless cleanup repeatedly fails. Log a non-secret diagnostic in development builds and retry deletion opportunistically.

Privacy and Security

  • All AI processing in V1 is cloud-based through the user's OpenRouter token.
  • The app must make cloud usage clear in Settings and in initial no-token states.
  • SecureStore holds the OpenRouter token. SQLite holds only non-secret settings and text history.
  • Audio should remain in app cache and should be deleted after result creation, cancel, or unrecoverable error cleanup.
  • Do not include provider request/response bodies in production logs.
  • Do not send saved history anywhere in V1 unless the user explicitly uses native Share or Copy into another app.

Testing Strategy

The implementation phase should establish a useful unit test suite before claiming the app is ready. Browser-level or simulator-level checks are not enough because most important behavior is orchestration, persistence, and error mapping.

Unit Tests

  • OpenRouter STT request builder: language mapping, Auto-detect omission, base64 field shape, model ID selection.
  • OpenRouter chat request builder: cleanup prompt, translation prompt, target language requirement, preset model mapping.
  • Provider error mapper: 400, 401, 402, 404, 429, 500, 502, 503, 524, 529, network errors, and malformed responses.
  • Recording state machine: idle, recording, stopping, processing, cancel, max-duration stop, retry states.
  • Flow orchestration: transcribe success, cleanup fallback, translation success, translation partial failure, manual text translation.
  • History repository: migrations, insert, edit, delete, tag creation, tag filtering, search, delete-all.
  • Settings repository: non-secret defaults and SecureStore token present/missing states through mocks.

Integration and Device Checks

  • Run the app on an iOS simulator for navigation, layout, settings, local storage, and mocked provider flows.
  • Run real microphone recording on a physical iPhone before readiness; simulator microphone behavior is not enough for acceptance.
  • Use short real samples in English, Spanish, and Arabic to benchmark STT quality, latency, translation quality, and cleanup behavior.
  • Verify audio files are deleted after success, cancel, and failed provider flows.
  • Verify native Copy and Share behavior on device.
  • Verify large text and Arabic text layout in result editor, history list, and detail view.

Builder and Verifier Method

After the user approves this design and then approves the implementation plan, development should use a builder/verifier loop. The builder implements a small slice. The verifier reviews it independently against the design, tests, and product behavior. Fixes repeat until the slice passes. This should happen before any Expo preview build or external distribution step.

Implementation Slices

This is not the implementation plan; it is the design-level breakdown that the later plan should expand into ordered tasks.

  1. Project scaffold and quality baseline after design and plan approval.
  2. Domain types, settings storage, token storage, and OpenRouter client tests.
  3. SQLite schema, migrations, history repository, tags, and search tests.
  4. Record tab mocked provider flow and result editor.
  5. Audio recording adapter, 60-second cap, temporary file cleanup, and physical-device checks.
  6. Live OpenRouter STT, cleanup, and translation provider integration.
  7. History tab, detail/edit/delete/tag flows, Copy and Share.
  8. Settings polish, no-token/permission/offline/provider error states.
  9. End-to-end verification, accessibility/layout pass, and readiness review.
