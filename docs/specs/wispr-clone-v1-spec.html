FlowLocal — your own Wispr Flow, owned & self-hosted

A personal macOS dictation app: press a hotkey, speak mostly English, and get intelligently cleaned-up text inserted at your cursor in any app. English is local-first; Arabic and Spanish are cloud-first language profiles with light cleanup. Your own dictionary, no subscription.

1 What v1 is (and isn't)

✓ In scope for v1

  • Global hotkey → record → cleaned text at cursor in any app
  • Language profiles from the menu bar: English local-first; Spanish/Arabic cloud-first
  • Switchable engines (local ⇄ cloud) still available for transcription & cleanup, but not the main daily workflow
  • OpenRouter model picker for cleanup (Claude / GPT / any)
  • Cleanup profiles: English medium cleanup; Arabic/Spanish light cleanup or pass-through
  • EN / ES / AR support, with English expected to be ~99% of daily use
  • Custom dictionary: name boosting + text-replacement shortcuts
  • Optional review & edit panel with a copy button
  • Privacy fix: allow temporary audio during capture/transcription, then delete it immediately after success/cancel unless debug audio retention is enabled (upstream keeps audio 7 days) + set transcript retention to 30 days (upstream default is 1 day)
  • Privacy toggle + simple cloud reminder: menu-bar cloud badge + one-time warning when switching from local to cloud

✗ Deferred (cheap to add later)

  • Spoken format commands ("new paragraph", "bullet")
  • Per-app tone modes (Slack casual vs email polished)
  • Auto-learn dictionary (app suggests new words)
  • Style preferences ("always JavaScript not javascript")
  • iOS / Android apps → v2
  • Windows → v4
  • Open-sourcing your fork → maybe, later

2 The pipeline (architecture)

One press-to-talk gesture drives a five-stage pipeline. v1 should use language-aware defaults: English stays local-first for speed/privacy; Spanish and Arabic route to cloud transcription and usually need only light cleanup.

Implementation note: VoiceInk records to an audio file for the current transcription path. v1 privacy behavior is therefore temporary audio only: create the file during capture/transcription, delete it immediately after success or cancel, and keep it only when debug audio retention is explicitly enabled. Only final text enters the 30-day history log.

English defaultLocal Whisper large-v3-turboOpenRouter Haiku-class medium cleanup by default; optional local/offline cleanup laterEnglish is expected to be ~99% of usage, so optimize for speed, privacy, and near-zero cost.
ArabicCloud STT: ElevenLabs Scribe v2 or OpenAI transcribeLight cleanup only: punctuation, spacing, obvious fixes; Sonnet/Haiku only when neededNo need to force local Arabic in v1. Cloud accuracy matters more, and Arabic dictation should not need aggressive rewriting.
SpanishCloud STT: ElevenLabs Scribe v2 or OpenAI transcribeLight cleanup only: punctuation, accents, obvious grammar; Sonnet/Haiku only when neededSpanish usage is occasional, so cloud mode keeps the local build smaller and avoids tuning local Spanish edge cases early.

Clarification: Haiku/Sonnet are text cleanup models, not speech-to-text engines. Arabic/Spanish still need a cloud transcription provider first; Haiku/Sonnet can then lightly normalize the transcript if useful.

3 Control scheme

Wispr-Flow-style, with your refinements. Two capture modes (you pick a default, both supported):

Push-to-talk (hold)

hold hotkey → speak → release → cleaned text inserts.

Fastest for short bursts. Uses the .onKeyDown/.onKeyUp primitives already in the fork.

Toggle (tap to start/stop)

tap hotkey → speak → Enter or tap hotkey → commit & insert.

Better for long dictation. The review panel fits naturally here.

KeyAction
Enter or fn+SpaceCommit — stop recording, run cleanup, insert at cursor
EscCancel — discard the dictation entirely (nothing inserted)
hotkey (configurable)Start / toggle recording
Open item — the fn key

Research could not confirm that macOS 26 lets an app intercept the fn/Globe key as a hotkey (it's reserved by the system; forum evidence says using it as a modifier is unreliable). Plan: ship with a rock-solid default combo (e.g. double-tap Right ⌥, or ⌃+Space) and attempt fn via a low-level event tap as a best-effort option. We verify this on day 1 of the build before committing to it.

Resource conclusion

Local English transcription is cheap enough to default on: large-v3-turbo is only ~1.5 GB on disk and ~1.8 GB RAM in the fork. Since Arabic/Spanish are cloud-first in v1, there is no need to install larger local STT models for multilingual edge cases. Local cleanup is still the heavier optional experiment: a good 27B/35B 4-bit LLM can consume ~14–22 GB and may miss the 2–3s target.

8 Roadmap beyond v1

v2 — iOS / Android (capture app)

Deliberately simple: one screen, mic button → speak → LLM-cleaned text in a box → Copy → paste anywhere. No system-wide insertion (impossible on iOS, messy on Android) — the copy-flow sidesteps the hardest mobile problem. Cloud-first thin client reusing the same engine layer; one cross-platform codebase.

v3 — Language modes (mobile)

Two modes on the mobile app: (1) Normal — concise, no-fluff transcription; (2) Learn-language — auto-detect source, translate into chosen target language(s) (EN→ES/AR), show both + transliteration + TTS playback. Later: conversation mode (2-speaker diarization). Full design → its own spec.

v4 — Windows

If you fork Handy instead (Tauri, cross-platform) the Windows path is nearly free — a reason to keep the cleanup layer engine-agnostic now so it can be reused.
