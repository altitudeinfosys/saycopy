Requirements interview

Mobile Transcription and Translation App

Durable capture for the product/design requirements interview. This file is the working source of truth for decisions that will later feed the MVP design doc and implementation plan.

Date: 2026-07-05 Status: In progress Artifact format: HTML

Goal

Define the MVP requirements for a separate mobile app, starting with iOS and later Android, that records speech, transcribes it, optionally translates it, and lets the user copy or share the result into other apps.

Summary / Key Decisions

  • The desktop VoiceInk/FlowLocal fork is no longer the primary product path; the user bought official VoiceInk for Mac.
  • The new product is a separate mobile transcription and translation app.
  • The initial platform preference is iOS first, with Android later if useful.
  • V1 should be built with Expo/React Native, not native Swift-only.
  • The pragmatic default is an Expo/React Native app with OpenRouter as the primary cloud AI provider.
  • The likely backend for a later synced-history version is Supabase, because it is fast for auth, database, storage, and React Native/Expo integration.
  • V1 should store transcription and translation history locally on the device. Database sync is deferred to v2.
  • V1 should include both Transcribe mode and Translate mode.
  • V1 Translate mode should support single-speaker translation only. Conversation mode is a future enhancement.
  • V1 should show English, Spanish, Arabic, and Auto-detect as first-class language choices.
  • V1 should use OpenRouter as the primary provider for speech-to-text, translation, and any cleanup/formatting because it centralizes API keys, model access, cost tracking, and dashboard visibility.
  • The app should recommend suitable OpenRouter models for translation, balancing translation quality, speed, latency, and cost.
  • V1 model selection should use simple presets such as Fast, Balanced, and Best Quality instead of exposing a full OpenRouter model picker in the primary UI.
  • The working product name is Tarek-Wisper, with final naming still open to alternatives.
  • OpenRouter currently documents a dedicated speech-to-text endpoint, /api/v1/audio/transcriptions, with base64 audio input and support for 99+ languages including English, Spanish, and Arabic.
  • All durable project artifacts should live in this project folder and be written as HTML unless the user explicitly requests otherwise.
  • This project folder has been initialized as a new Git repository on main.

Q&A Log

Q1 - Project Setup and Artifact Format

Asked: Should this folder be initialized as the project Git repository, and should all artifacts be HTML?

Captured: The user confirmed that the folder should be initialized as a Git repository. The user also requested that all artifacts be HTML and that this rule be recorded in AGENTS.md.

Flags: A remote has not been configured yet, so upstream sync is not applicable until a remote is added.

Q2 - Accounts and History Storage

Asked: Should v1 require accounts and synced history from day one, or should it start account-free with local-only history and add Supabase sync later?

Captured: V1 should store history locally on the device. Account-based database sync can be considered for v2.

Flags: V2 backend choice remains open, with Supabase currently the likely default.

Q3 - MVP Modes

Asked: Should v1 support both Transcribe and Translate modes, or focus only on Transcribe first and add Translate immediately after?

Captured: V1 should include both Transcribe and Translate modes.

Flags: None.

Q4 - App Framework

Asked: Should v1 be built as Expo/React Native so Android is easier later, or as native Swift iOS for the most iOS-specific app first?

Captured: V1 should use Expo/React Native.

Flags: None.

Q5 - Primary AI Provider

Asked: Should v1 use one cloud AI provider wherever possible for simplicity, or separate best-in-class providers for transcription and translation?

Captured: The user prefers OpenRouter as the primary provider because it gives access to many AI models and keeps tracking in one dashboard. Current OpenRouter documentation supports this direction: it documents speech-to-text via /api/v1/audio/transcriptions, base64 audio input, auto language detection when the language parameter is omitted, and speech-to-text support across 99+ languages including English, Spanish, and Arabic. Translation can be handled through OpenRouter text models.

Flags: Keep a small provider abstraction so a later fallback provider can be added if OpenRouter transcription quality, latency, file limits, or pricing is not good enough for real mobile use.

Q6 - Working Product Name

Asked: What working app name should be used in the docs and design?

Captured: The working product name is Tarek-Wisper. The user is also open to other name suggestions, so final branding can be revisited later.

Flags: Include a short naming options section in the design doc or a later branding note.

Q7 - Translation Interaction Scope

Asked: In Translate mode, should v1 support only single-speaker translation or also a conversation mode where two people speak back and forth?

Captured: V1 should support single-speaker translation only. Conversation mode should be documented as a future enhancement.

Flags: Conversation mode belongs in the future roadmap, not the v1 MVP.

Q8 - First-Class Languages and Model Suggestions

Asked: Which languages should be shown as first-class choices in v1?

Captured: V1 should show English, Spanish, Arabic, and Auto-detect as first-class language choices. The user also wants the app to offer model suggestions for translation tasks, with a preference for efficient, easy, fast, low-latency choices.

Flags: Exact OpenRouter model recommendations are time-sensitive and should be verified against current OpenRouter model availability, pricing, and latency data during design and implementation.

Q9 - Model Selection UI

Asked: Should model selection be exposed as simple presets like Fast, Balanced, and Best Quality, or should users choose from a full OpenRouter model picker?

Captured: The user agreed with simple v1 presets such as Fast, Balanced, and Best Quality.

Flags: A full OpenRouter model picker can be considered later as an advanced setting.

Open Flags

  • For v2, decide whether synced history uses Supabase, Neon plus custom auth/API, or another backend.
  • Revisit final product naming; Tarek-Wisper is the working name for now.
  • Future enhancement: add conversation mode for two-person back-and-forth translation.
  • Benchmark OpenRouter speech-to-text quality and latency on real English, Spanish, and Arabic samples during implementation.
  • Verify current OpenRouter model options before finalizing default translation model recommendations.
  • Future enhancement: consider an advanced OpenRouter model picker after the preset flow is working.
  • Confirm the first set of MVP screens.
  • Add a Git remote before future implementation work that depends on upstream sync.
