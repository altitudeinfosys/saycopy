Requirements interview

Mobile Transcription and Translation App

Durable capture for the product/design requirements interview. This file is the working source of truth for decisions that will later feed the MVP design doc and implementation plan.

Date: 2026-07-05 Status: In progress Artifact format: HTML

Goal

Define the MVP requirements for a separate mobile app, starting with iOS and later Android, that records speech, transcribes it, optionally translates it, and lets the user copy or share the result into other apps.

Summary / Key Decisions

  • The desktop VoiceInk/FlowLocal fork is no longer the primary product path; the user bought official VoiceInk for Mac.
  • The new product is a separate mobile transcription and translation app.
  • The initial platform preference is iOS first, with Android later if useful.
  • V1 should be built with Expo/React Native, not native Swift-only.
  • The pragmatic default is an Expo/React Native app with OpenRouter as the primary cloud AI provider.
  • The likely backend for a later synced-history version is Supabase, because it is fast for auth, database, storage, and React Native/Expo integration.
  • V1 should store transcription and translation history locally on the device. Database sync is deferred to v2.
  • V1 should include both Transcribe mode and Translate mode.
  • V1 Translate mode should support single-speaker translation only. Conversation mode is a future enhancement.
  • V1 should show English, Spanish, Arabic, and Auto-detect as first-class language choices.
  • V1 should use OpenRouter as the primary provider for speech-to-text, translation, and any cleanup/formatting because it centralizes API keys, model access, cost tracking, and dashboard visibility.
  • The app should recommend suitable OpenRouter models for translation, balancing translation quality, speed, latency, and cost.
  • V1 model selection should use simple presets such as Fast, Balanced, and Best Quality instead of exposing a full OpenRouter model picker in the primary UI.
  • Transcribe mode should apply light cleanup by default, including punctuation, capitalization, and modest filler-word cleanup.
  • V1 should include a Settings page where the user can enter an OpenRouter API token, stored locally and securely on the device. The app should not embed a shared production API token.
  • V1 should save text outputs in history, not audio recordings. Audio should be temporary during recording/transcription and deleted after the text result is created.
  • V1 should include simple optional tags for saved history items, with History filtering by tag.
  • V1 should use three main tabs: Record, History, and Settings.
  • Successful transcription and translation results should auto-save to History by default, and history items should be deletable.
  • Result text should be editable before copy/share, and saved history should reflect the user's edited text.
  • History should include basic text search in addition to chronological browsing and tag filtering.
  • Translate mode should show both the original transcription and the translated output, with the translation emphasized.
  • V1 should include both Copy and native Share actions for result text.
  • Translate mode should support simple manual text input in addition to voice-to-translation.
  • Voice recording should use tap once to start and tap again to stop for v1.
  • V1 should be cloud-only for transcription, translation, and cleanup. Offline/local speech models are deferred.
  • V1 should optimize for short recording clips, around 60 seconds max, rather than long dictation with audio chunking.
  • The approved product direction is Approach A: Focused Cloud MVP.
  • The working product name is Tarek-Wisper, with final naming still open to alternatives.
  • OpenRouter currently documents a dedicated speech-to-text endpoint, /api/v1/audio/transcriptions, with base64 audio input and support for 99+ languages including English, Spanish, and Arabic.
  • All durable project artifacts should live in this project folder and be written as HTML unless the user explicitly requests otherwise.
  • This project folder has been initialized as a new Git repository on main.

Q&A Log

Q1 - Project Setup and Artifact Format

Asked: Should this folder be initialized as the project Git repository, and should all artifacts be HTML?

Captured: The user confirmed that the folder should be initialized as a Git repository. The user also requested that all artifacts be HTML and that this rule be recorded in AGENTS.md.

Flags: A remote has not been configured yet, so upstream sync is not applicable until a remote is added.

Q2 - Accounts and History Storage

Asked: Should v1 require accounts and synced history from day one, or should it start account-free with local-only history and add Supabase sync later?

Captured: V1 should store history locally on the device. Account-based database sync can be considered for v2.

Flags: V2 backend choice remains open, with Supabase currently the likely default.

Q3 - MVP Modes

Asked: Should v1 support both Transcribe and Translate modes, or focus only on Transcribe first and add Translate immediately after?

Captured: V1 should include both Transcribe and Translate modes.

Flags: None.

Q4 - App Framework

Asked: Should v1 be built as Expo/React Native so Android is easier later, or as native Swift iOS for the most iOS-specific app first?

Captured: V1 should use Expo/React Native.

Flags: None.

Q5 - Primary AI Provider

Asked: Should v1 use one cloud AI provider wherever possible for simplicity, or separate best-in-class providers for transcription and translation?

Captured: The user prefers OpenRouter as the primary provider because it gives access to many AI models and keeps tracking in one dashboard. Current OpenRouter documentation supports this direction: it documents speech-to-text via /api/v1/audio/transcriptions, base64 audio input, auto language detection when the language parameter is omitted, and speech-to-text support across 99+ languages including English, Spanish, and Arabic. Translation can be handled through OpenRouter text models.

Flags: Keep a small provider abstraction so a later fallback provider can be added if OpenRouter transcription quality, latency, file limits, or pricing is not good enough for real mobile use.

Q6 - Working Product Name

Asked: What working app name should be used in the docs and design?

Captured: The working product name is Tarek-Wisper. The user is also open to other name suggestions, so final branding can be revisited later.

Flags: Include a short naming options section in the design doc or a later branding note.

Q7 - Translation Interaction Scope

Asked: In Translate mode, should v1 support only single-speaker translation or also a conversation mode where two people speak back and forth?

Captured: V1 should support single-speaker translation only. Conversation mode should be documented as a future enhancement.

Flags: Conversation mode belongs in the future roadmap, not the v1 MVP.

Q8 - First-Class Languages and Model Suggestions

Asked: Which languages should be shown as first-class choices in v1?

Captured: V1 should show English, Spanish, Arabic, and Auto-detect as first-class language choices. The user also wants the app to offer model suggestions for translation tasks, with a preference for efficient, easy, fast, low-latency choices.

Flags: Exact OpenRouter model recommendations are time-sensitive and should be verified against current OpenRouter model availability, pricing, and latency data during design and implementation.

Q9 - Model Selection UI

Asked: Should model selection be exposed as simple presets like Fast, Balanced, and Best Quality, or should users choose from a full OpenRouter model picker?

Captured: The user agreed with simple v1 presets such as Fast, Balanced, and Best Quality.

Flags: A full OpenRouter model picker can be considered later as an advanced setting.

Q10 - Transcription Cleanup Default

Asked: In Transcribe mode, should the default output be a mostly raw transcript, or should it be lightly cleaned up with punctuation, capitalization, and filler-word cleanup?

Captured: The user agreed with light cleanup by default.

Flags: The design should decide whether to store raw transcript text internally when available or only store the cleaned output in v1.

Q11 - API Key Handling

Asked: Is v1 a personal app using the user's own OpenRouter API key, or should it be designed for multiple users with each user bringing their own key?

Captured: V1 should include a Settings page where the user can enter an OpenRouter token. The token should be stored locally and securely on the device. The app should not embed a shared production OpenRouter token in the shipped mobile app.

Flags: The design should specify the secure local token storage mechanism for Expo/React Native.

Q12 - Audio Retention and Tags

Asked: Should v1 save audio recordings in history, or only save the text outputs?

Captured: The user agreed with saving text only. Audio should be temporary during recording/transcription and deleted after the result is created. The user also wants the ability to add tags or categories to saved items so history can be retrieved quickly and easily later.

Flags: Decide whether tagging belongs in v1 or v2.

Q13 - V1 Tagging Scope

Asked: Should simple tags be included in v1?

Captured: The user confirmed that simple tags should be part of v1. Scope should stay lightweight: optional user-defined tags on saved transcript and translation items, plus filtering by tag in History.

Flags: Do not include nested folders, complex category management, or AI auto-tagging in the first pass unless later approved.

Q14 - Primary Navigation

Asked: Should the main app navigation be three tabs: Record, History, and Settings?

Captured: The user agreed with three main tabs. Transcribe and Translate should live inside Record, saved text and tags should live in History, and API token/model/language defaults should live in Settings.

Flags: None.

Q15 - History Save and Delete Behavior

Asked: After a successful transcription or translation, should the app auto-save the result to History, or should saving be manual?

Captured: Successful transcription and translation results should auto-save to History by default. History items should also support deletion.

Flags: None.

Q16 - Result Editing

Asked: Should the result text be editable before copy/share, or should v1 show read-only output with copy/share only?

Captured: Result text should be editable before copy/share. Saved history should reflect the user's edited text.

Flags: Decide whether raw/original AI output is stored separately for troubleshooting or omitted in v1.

Q17 - Translate Result Display

Asked: For Translate mode, should the app show both the original transcription and the translated output, or only the translated output?

Captured: Translate mode should show both the original transcription and the translated output. The translated output should be emphasized, and saved history items should keep both fields.

Flags: None.

Q18 - Copy and Share Actions

Asked: Should v1 include Share using the native iOS share sheet in addition to Copy?

Captured: V1 should include both Copy and native Share actions for result text.

Flags: None.

Q19 - Manual Text Translation

Asked: Should v1 support manual text input for translation too, or only voice-to-translation?

Captured: V1 should support simple manual text input in Translate mode, in addition to voice-to-translation.

Flags: Keep this lightweight in v1; it should not become a separate document editor.

Q20 - Recording Interaction

Asked: Should recording be tap once to start and tap again to stop, or press and hold to record?

Captured: Voice recording should use tap once to start and tap again to stop for v1.

Flags: Press-and-hold can be considered later as an alternate short-clip mode.

Q21 - Offline Support

Asked: Should v1 require an internet connection for all transcription and translation, or should it include offline/local speech features?

Captured: V1 should be cloud-only for transcription, translation, and cleanup. Offline/local speech models are deferred.

Flags: The design should include clear offline and provider-error states.

Q22 - History Search

Asked: Should v1 include search in History, or only tag filtering and chronological browsing?

Captured: V1 should include basic text search in History, in addition to tag filtering and chronological browsing.

Flags: None.

Q23 - Recording Length

Asked: Should v1 keep recordings to short clips, around 60 seconds max, or support longer dictation by splitting audio into chunks?

Captured: V1 should optimize for short recording clips, around 60 seconds max.

Flags: Longer dictation with audio chunking belongs in the future roadmap.

Q24 - Approach Approval

Asked: Is Approach A, the Focused Cloud MVP, approved as the direction?

Captured: The user approved moving forward with Approach A: Expo/React Native, OpenRouter primary provider, local text history, simple tags/search, no accounts, no offline mode, and no long-audio chunking in v1.

Flags: None.

Open Flags

  • For v2, decide whether synced history uses Supabase, Neon plus custom auth/API, or another backend.
  • Revisit final product naming; Tarek-Wisper is the working name for now.
  • Future enhancement: add conversation mode for two-person back-and-forth translation.
  • Benchmark OpenRouter speech-to-text quality and latency on real English, Spanish, and Arabic samples during implementation.
  • Verify current OpenRouter model options before finalizing default translation model recommendations.
  • Future enhancement: consider an advanced OpenRouter model picker after the preset flow is working.
  • Decide whether v1 stores raw transcript text internally when available, or only the cleaned output shown to the user.
  • Specify the Expo/React Native secure local storage mechanism for the OpenRouter token.
  • Future enhancement: consider nested folders, complex category management, or AI auto-tagging after simple manual tags are working.
  • Decide whether raw/original AI output is stored separately for troubleshooting or omitted in v1.
  • Include offline and provider-error states in the design.
  • Future enhancement: support longer dictation by splitting audio into chunks and merging results.
  • Add a Git remote before future implementation work that depends on upstream sync.
