Tarek-Wisper V1 Implementation Plan

This plan builds the approved Focused Cloud MVP: an iOS-first Expo/React Native app for short cloud transcription, single-speaker translation, local text history, search, tags, copy/share, and secure user-provided OpenRouter token storage.

Gate

No app scaffold, package install, source code, external preview build, EAS build, TestFlight build, or similar external build step should start until the user approves this implementation plan. Before implementation begins, run git fetch origin main and fast-forward local main from origin/main per AGENTS.md.

For agentic workers: required sub-skill during execution is superpowers:subagent-driven-development. The execution loop is builder subagent, verifier subagent, fix subagent, verifier subagent, repeated until all critical and high issues are resolved or a true external blocker is reached.

How You Will Test It Once Built

The fastest practical path is local Expo development first, then a physical iPhone smoke test. This does not require an Expo preview build if the selected Expo modules run in Expo Go.

  1. Run the automated suite: npm test, npm run typecheck, npm run lint, and npx expo-doctor.
  2. Start the local dev server: npx expo start.
  3. Open in the iOS simulator from the Expo CLI with i, or run npx expo start --ios.
  4. For real microphone testing, install Expo Go on your iPhone, run npx expo start --tunnel or LAN mode, scan the QR code, and enter your OpenRouter token in Settings.
  5. Run acceptance flows: Transcribe English, Translate manual text, Translate voice to Spanish/Arabic/English, edit result, Copy, Share, tag, search, delete.
  6. Before any EAS preview build, TestFlight build, or external installable build, stop and ask the user for approval.

If Expo Go cannot run a required native module or configuration, the next option is a development build. That is an external build step, so the agent must stop and ask first.

UI Direction

Build the agreed Option 1.5: the clean native light layout from the first mockup, with the stronger active-recording panel and saved-history status treatment from the third mockup.

  • Default theme: light, calm, iOS-native, utility-first.
  • Record tab: simple segmented mode switch, language and preset controls, large tap-to-record control, editable result card.
  • Borrow from warm premium mockup: active recording panel with timer/waveform/progress feel, saved-to-history row, slightly stronger button polish.
  • Do not use the immersive dark assistant look as V1 default. It can become a future theme.

Execution Protocol

Severity Rules

  • Critical: app cannot launch, secrets leak, audio persists against policy, destructive history bug, or live provider flow unusable.
  • High: required V1 flow broken, tests missing for shared logic, no-token/error states misleading, major layout overlap, Arabic unusable.
  • Medium: polish issue, non-blocking UX friction, missing nonessential empty state, minor code organization concern.
  • Low: wording, visual tuning, cleanup suggestions that do not affect correctness.

Builder/Verifier Loop

  1. Builder subagent gets exactly one task slice and the relevant source files only.
  2. Builder writes or updates tests first where the slice includes logic.
  3. Builder implements the smallest complete slice and reports commands run.
  4. Verifier subagent independently reviews the slice against this plan and the design doc.
  5. Verifier classifies issues by severity and requests fixes for all critical/high issues.
  6. Fix builder handles only verifier findings, then verifier rechecks.
  7. Main agent accepts the slice only after critical/high issues are resolved and tests for that slice pass.
